## 核心内容摘要

这篇文章系统解释了“上下文工程”为什么正在替代单纯的 prompt engineering，成为 LLM 应用效果的关键分水岭。作者先从 transformer 的注意力机制讲起，指出模型并不是线性阅读输入，而是对全体 token 做关系计算，因此长上下文会同时带来计算成本上升、注意力分布不均和“中间遗忘”问题。进一步地，文章引用 Chroma 的研究说明，随着输入变长，模型准确率并不会平滑下降，而是可能在某个阈值后突然崩塌，这种现象被称为 context rot。基于这些结构性约束，作者把上下文工程定义为：在每次调用前，对系统指令、用户输入、会话历史、检索知识、工具描述和工具输出这整套信息环境进行设计、组装和管理。围绕这一目标，文中总结了四种核心策略：把信息写入外部存储、只选择最相关内容、对历史和输出做压缩，以及通过多 agent 把不同任务隔离到不同上下文窗口中。全文的核心观点很明确：很多 LLM 失败并不是模型不够聪明，而是喂给模型的上下文不对。

## 关键要点

- **更多上下文不等于更好结果**：LLM 的注意力并不会均匀覆盖整段输入，开头和结尾通常更受关注，中间信息更容易被忽略。
- **长上下文存在突发式性能退化**：context rot 说明模型在超过某个长度阈值后可能突然掉准，而不是随输入长度线性变差。
- **上下文工程关注的是整个信息系统**：它不只是在打磨一句提示词，而是在决定这次调用到底该看到哪些指令、历史、检索内容与工具结果。
- **四种主策略分别对应不同约束**：`write` 解决无状态和窗口有限，`select` 解决相关性筛选，`compress` 解决上下文膨胀，`isolate` 解决多种信息混杂导致的干扰。
- **RAG、多 agent 和记忆系统本质上都是上下文管理手段**：它们的价值不只是“接外部能力”，而是控制模型当下能看到什么、看不到什么。
- **工程权衡无法回避**：压缩会丢细节，检索会带来噪声，多 agent 会提高成本和复杂度，核心是针对任务找到合适的信息密度与组织方式。

## 术语翻译

- **Context Engineering**：上下文工程
- **Context Window**：上下文窗口
- **Attention**：注意力机制
- **Context Rot**：上下文腐化
- **Lost in the Middle**：中间遗忘

这里把 `context rot` 译为“上下文腐化”，是为了强调它不是单纯的长度变长，而是随着输入堆积，模型对重要信息的辨识能力被噪声逐步侵蚀。

## 我的思考与感悟

这篇文章最有价值的地方，是把很多零散工程实践拉回到一个统一框架里：memory、RAG、tool selection、history compaction、多 agent，其实都属于 context engineering。这个视角很适合拿来审视 agent 系统，因为实际效果往往不是输在模型选型，而是输在“这一轮到底给了模型什么”。对我来说，做知识归档这类任务本身也很像在练上下文工程：不是把看到的内容全塞进去，而是保留真正影响理解与检索的那部分信息。
